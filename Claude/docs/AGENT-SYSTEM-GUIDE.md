## 내가 원하는 산출물

최종적으로 **하나의 통합 설계서 문서(md 파일)**를 만들고 싶다.
이 문서는 Claude Code에서 구체적인 구현을 할 때 참조할 **계획서** 역할을 한다.

> **⚠️ 범위**: 이 문서는 **설계서(계획서)**이다. CLAUDE.md, 에이전트 파일, 스킬 파일의 **상세 내용은 작성하지 않는다.** 구현 스펙은 **구조와 역할 정의 수준**까지만 포함하며, 실제 내용은 구현 단계에서 작성한다.

### 설계서에 포함될 내용

1. **작업 컨텍스트 문서**
   - 배경, 목적, 범위, 입출력 정의, 제약조건, 용어 정의

2. **워크플로우 정의**
   - 단계별 흐름도, 분기조건, 상태 전이
   - LLM 판단 영역과 코드 처리 영역 구분
   - LLM이 직접 수행할 판단/의사결정 단계 명시
   - 각 단계의 성공 기준 및 검증 방법
   - 실패 시 처리 방식 (재시도 / 에스컬레이션 / 스킵)
   - **승인 게이트**: 단계 간 사용자 확인이 필요한 지점과 처리 방식

3. **구현 스펙** (구조 개요만, 상세 내용 X)
   - 폴더 구조
   - CLAUDE.md 핵심 섹션 목록
   - 에이전트 구조 (단일 또는 서브에이전트)
   - 작업 단계별 처리 방식 (에이전트 판단 vs 스크립트)
   - 스킬/스크립트 파일 목록, 역할, 트리거 조건(언제 이 스킬이 호출되는가)
   - 서브에이전트 사용 시: 이름, 역할, 입출력, 데이터 전달 방식, **모델 선택**
   - 주요 산출물 파일 형식

## 설계 원칙

### [폴더 구조]

에이전트 설정은 `.claude/` 하위에 배치한다.

```
/project-root
  ├── CLAUDE.md                        # 메인 에이전트 지침 (오케스트레이터)
  ├── .claude/
  │   ├── skills/<skill-name>/         # 스킬: 디렉토리 안에 .md 파일 배치
  │   │   ├── guide.md                 # 스킬 지침 (파일명 자유)
  │   │   └── reference.md             # (선택) 도메인 지식, 카탈로그 등
  │   └── agents/
  │       ├── <agent-name>.md          # 서브에이전트: 파일명 = 에이전트 이름
  │       └── <agent-name-2>.md
  ├── output/                          # 산출물
  └── docs/                            # (선택) 참고 문서
```

> **참고**: 에이전트는 디렉토리가 아닌 **단일 `.md` 파일**로 정의한다 (`<agent-name>.md`). 스킬은 **디렉토리** 안에 `.md` 파일들을 배치한다. 스크립트(결정론적 도구)는 프로젝트 루트나 별도 위치에 두고, 에이전트/스킬의 시스템 프롬프트에서 경로를 참조하는 방식을 권장한다.

### [판단과 코드의 역할 분리]

| 에이전트가 직접 수행 | 스크립트로 처리 |
|---------------------|----------------|
| 분류, 의사결정, 우선순위 판단 | 파일 I/O, 데이터 파싱 |
| 품질 평가, 정성적 분석 | 외부 API 호출 |
| 컨텍스트 기반 추론 | 반복 처리, 집계 |
| 자연어 생성/요약 | 정적 분석, 테스트 실행 |

### [검증 패턴]

각 워크플로우 단계에는 반드시 성공 기준을 정의한다. 검증 유형은 산출물 성격에 따라 선택한다.

| 검증 유형 | 적용 대상 | 예시 |
|-----------|----------|------|
| **스키마 검증** | 구조화된 산출물 | 필수 필드 존재, 타입 체크 |
| **규칙 기반** | 정량 기준이 있는 작업 | 항목 수, 글자 수, 필수 섹션 포함 |
| **LLM 자기 검증** | 정성적 판단 작업 | 요약 품질, 톤 적합성, 누락 여부 |
| **사람 검토** | 고위험 최종 산출물 | 외부 발송 문서, 의사결정 |

설계서 작성 시 각 단계마다 다음을 명시한다:
- **성공 기준**: 이 단계의 산출물이 충족해야 할 조건
- **검증 방법**: 위 유형 중 어떤 방식으로 확인하는가
- **실패 시 처리**: 아래 패턴 중 선택

### [승인 게이트]

워크플로우 중간에 사용자 확인이 필요한 지점을 **승인 게이트**로 정의한다. 검증 패턴의 "사람 검토"와 달리, 승인 게이트는 **다음 단계 진행 여부**를 결정하는 흐름 제어 장치이다.

| 요소 | 설명 |
|------|------|
| **게이트 위치** | 어떤 단계와 단계 사이에 배치하는가 |
| **제시 형식** | 사용자에게 보여줄 요약 포맷 (핵심 항목만 간결하게) |
| **응답 분기** | 승인 → 다음 단계 / 수정 요청 → 해당 단계 재실행 후 게이트 복귀 |

설계 시 고려사항:
- 모든 단계에 게이트를 넣으면 흐름이 느려진다 → **되돌리기 어려운 단계 앞**에만 배치
- 게이트에서 보여줄 요약은 전체 산출물이 아니라 **핵심 의사결정 사항**만 포함
- 예: 기획안 승인, 구조 설계 승인, 최종 산출물 승인

### [실패 처리]

| 방식 | 사용 시점 |
|------|----------|
| **자동 재시도** | 검증 실패가 단순 누락/형식 오류인 경우 (최대 재시도 횟수 명시) |
| **에스컬레이션** | 판단 불확실성이 높거나 기준 자체가 모호한 경우 → 사람에게 확인 요청 |
| **스킵 + 로그** | 선택적 단계이며 전체 흐름에 영향 없는 경우 → 사유를 로그에 기록 |

### [에이전트 구조 선택]

**단일 에이전트** (기본):
- 워크플로우가 단순하고 지침이 짧을 때

**서브에이전트 분리** (필요 시):
- **컨텍스트 윈도우 최적화가 필요할 때** -- 지침이 길어서 항상 로드하면 비효율적인 경우, 서브에이전트로 분리하여 필요한 시점에만 해당 지침 로드
- 독립적인 작업 블록이 명확히 구분되고, 각각 다른 도메인 지식이 필요할 때

### [모델 선택]

서브에이전트마다 작업 특성에 맞는 모델을 지정할 수 있다. 설계 시 비용/성능 트레이드오프를 고려한다.

| 모델 | 적합한 작업 | 비용 |
|------|------------|------|
| **opus** | 복잡한 판단, 창의적 생성, 다단계 추론 | 높음 |
| **sonnet** | 대부분의 생성/분석 작업 (기본값) | 중간 |
| **haiku** | 단순 변환, 템플릿 기반 생성, 포맷팅 | 낮음 |

원칙:
- 기본값은 **sonnet** — 대부분의 에이전트에 적합
- 핵심 판단/기획처럼 품질이 전체 파이프라인에 영향을 미치는 단계 → **opus** 고려
- 정해진 템플릿을 채우거나 간단한 변환 작업 → **haiku**로 비용 절감
- 설계서에 각 에이전트의 모델 선택 이유를 한 줄로 명시

### [서브에이전트 설계]

- 메인 에이전트(CLAUDE.md)가 오케스트레이터 역할
- 서브에이전트 간 직접 호출 금지 → 메인을 통해 조율
- 에이전트 파일(`<agent-name>.md`)에 역할, 트리거 조건, 입출력, 참조 스킬 명시

### [데이터 전달 패턴]

| 방식 | 사용 시점 |
|------|----------|
| **파일 기반** | 데이터가 크거나 구조화된 경우 → `/output/step1_result.json` |
| **프롬프트 인라인** | 메타데이터 수준의 짧은 값 (수십 줄 이내) |

권장: 중간 산출물은 `/output/`에 저장하고 파일 경로만 전달

> **주의**: 프롬프트 인라인은 서브에이전트의 컨텍스트 윈도우를 소모한다. 본문 콘텐츠는 반드시 파일 기반으로 전달하고, 프롬프트에는 파일 경로 + 간단한 메타데이터만 포함한다.

### [스킬 vs 서브에이전트]

| 스킬 | 서브에이전트 |
|------|-------------|
| 도구/기능 단위 (작음) | 역할/책임 단위 (큼) |
| 여러 에이전트가 공유 가능 | 특정 워크플로우 전용 |
| 예: `file-parser`, `api-caller` | 예: `code-reviewer`, `report-generator` |

## 진행 방식

### 1단계: 작업 파악 (필수 확인 항목)

사용자 입력을 받으면 아래 세 영역의 정보가 충분한지 판단한다.
**부족한 영역이 있으면 해당 영역의 질문만 한다. 모두 충분하면 바로 설계에 착수한다.**

| 영역 | 확인할 것 | 부족할 때 질문 예시 |
|------|----------|-------------------|
| **작업 절차** | 입력→출력까지의 구체적 단계가 있는가? 분기 조건이 명확한가? | "A 이후에 B로 갈지 C로 갈지는 어떤 기준으로 판단하나요?" |
| **에이전트 조직** | 단일/멀티 에이전트 선호가 있는가? 역할 분리 기준이 있는가? | "이 작업을 하나의 에이전트가 순차 처리하면 되나요, 아니면 분리하고 싶은 역할이 있나요?" |
| **기술 스택** | 사용할 도구, API, 라이브러리, 파일 형식 등이 있는가? | "데이터 소스는 뭔가요? 특정 API나 도구를 이미 쓰고 있나요?" |

### 질문 원칙

- 한 번에 최대 3개까지만 질문한다
- 사용자가 모르겠다고 하면 설계 원칙에 따라 LLM이 판단하고 그 근거를 명시한다
- 사용자가 "알아서 해줘"라고 하면, 합리적 기본값을 적용하되 핵심 선택지만 확인한다

### 2단계: 설계서 작성

확인이 끝나면 "설계서에 포함될 내용"에 따라 통합 설계서(md 파일)를 작성한다.

### 3단계: 리뷰

설계서 초안을 제시하고, 수정할 부분이 있는지 확인한다.
